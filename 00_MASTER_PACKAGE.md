# 00_START_HERE

## What RepoBrain is

RepoBrain is a codebase intelligence product. It connects to a GitHub repository, indexes it deeply (files, symbols, dependencies, structure), and provides a persistent intelligence layer over the code. Users ask questions and get grounded, cited answers. They inspect files with context. They receive improvement recommendations. Eventually, they propose, preview, and apply changes through controlled Git workflows.

The mental model: **your codebase has a brain, and you're talking to it.**

## What to build first

1. **Foundation** — Auth, workspace model, database schema, project skeleton
2. **GitHub connection** — OAuth App, clone repo, store metadata
3. **Ingestion pipeline** — Parse files, extract symbols (tree-sitter), chunk, embed, store
4. **Retrieval engine** — Semantic + lexical + structural search, context assembly
5. **Q&A with citations** — LLM orchestration, grounded answers, file:line references
6. **Workspace UI** — File tree, code viewer, chat pane, citation links

That is v1. It must feel like a product, not a demo.

## What must NOT be built yet

- Multi-repo workspaces (Phase 4)
- Change proposal / apply / commit / push (Phase 3)
- PR generation (Phase 4)
- Real-time webhook sync on push (Phase 2)
- Custom embedding model training
- IDE plugins or extensions
- Autonomous agent behavior

## How to use this package

Read sections 00–15 in order. Each section builds on the previous. The implementation plan (§10) gives the exact build order. The module boundaries (§09) define what each component owns. The acceptance criteria (§11) define when each piece is done.

**Treat this as a contract.** Deviate only when explicitly justified and flag the deviation.

## Critical invariants

1. Every answer must cite specific files and line ranges. No uncited claims.
2. The retrieval pipeline must combine semantic + lexical + structural signals. No single-strategy retrieval.
3. The domain model must support multi-repo from day one in schema, even though v1 only implements single-repo.
4. The change workflow must never blur "suggested" and "applied". These are distinct states with explicit user transitions.
5. The ingestion pipeline must be idempotent. Re-indexing the same commit must produce identical results.

# 01_PRODUCT_BRIEF

## Target user

Senior developers, tech leads, and engineering managers who work with codebases they did not write or have partially forgotten. Developers joining new teams. Technical due diligence reviewers. Anyone who needs to understand a codebase faster and more deeply than reading files one at a time.

## Core user jobs

1. **Understand** — "How does authentication work in this repo?" → Get a grounded architectural answer with file references.
2. **Navigate** — "Show me everything related to payment processing" → Get a curated, relevant file list with context.
3. **Assess** — "What are the biggest technical debt items?" → Get specific, cited recommendations.
4. **Inspect** — Click a citation → See the exact code with surrounding context.
5. **Improve** — "How should I refactor the auth middleware?" → Get a concrete, grounded suggestion.
6. **Change** (Phase 3) — "Apply this refactor" → Preview diff, approve, branch, commit, push.

## Primary product promise

**You will understand any codebase as if you wrote it, in minutes instead of weeks.**

## Key differentiators

- **Grounded, not generic.** Every answer references actual files, symbols, and line ranges in your repo. No hallucinated APIs.
- **Persistent understanding.** The index persists. The system learns your repo's structure, not just individual files.
- **Third-person intelligence.** RepoBrain reasons about the codebase from the outside — architecture, dependencies, patterns, debt — not just autocompleting inside a file.
- **Controlled change.** When changes happen, they go through explicit propose → preview → approve → apply → commit → push. Never silent mutations.

## What "third-person intelligence layer" means practically

First-person tools (Copilot, Cursor) help you write code while you're inside a file. Third-person tools help you understand, assess, and reason about code from the outside. RepoBrain is third-person: it sees the whole repo, understands relationships between components, and gives you the kind of perspective a senior architect would after studying the codebase for a week.

## Anti-goals

- **Not an IDE.** RepoBrain does not replace VS Code or Cursor. It complements them.
- **Not Devin.** RepoBrain does not autonomously plan and execute multi-step engineering tasks.
- **Not GitHub Copilot.** RepoBrain does not do inline code completion.
- **Not a generic chatbot.** RepoBrain must refuse to answer questions it cannot ground in the codebase.
- **Not a monitoring tool.** RepoBrain does not watch for runtime errors or performance issues.

# 02_VISION_AND_PHASE_STRATEGY

## Full product vision

RepoBrain becomes the standard way engineering teams understand and manage their codebases. Connect your repos, get instant deep understanding, ask any question, get grounded answers, and safely apply improvements — all from one workspace. Eventually, RepoBrain handles multi-repo reasoning (how does service A depend on service B?), generates PRs from improvement suggestions, and provides team-wide codebase health dashboards.

## Phase model

### Phase 0: Foundation (Week 1–2)
- Project skeleton (Next.js monorepo)
- Auth (GitHub OAuth)
- Workspace + repo connection model
- Database schema (Postgres + pgvector)
- Background job infrastructure (BullMQ + Redis)
- UI shell (layout, navigation, empty states)

### Phase 1: Index + Understand (Week 3–5)
- Clone repo to server storage
- File ingestion: parse all files, detect language, extract metadata
- Symbol extraction via tree-sitter (functions, classes, exports, imports)
- Chunking strategy (symbol-aware boundaries)
- Embedding generation (OpenAI text-embedding-3-small or equivalent)
- pgvector storage + HNSW index
- Lexical index (Postgres full-text search)
- Symbol graph (imports/exports/call relations)
- Repo summary generation (LLM-produced architecture overview)
- Basic retrieval: semantic + lexical + structural
- Q&A endpoint with citation generation
- File tree in UI
- Chat pane with citation rendering

### Phase 2: Inspect + Improve (Week 6–8)
- Code viewer with syntax highlighting
- Citation click → jump to file:line
- Architecture summary view
- Improvement recommendations engine
- Technical debt analysis
- File-scoped Q&A (ask about a specific file with full context)
- Re-index on demand
- Webhook-triggered re-index on push

### Phase 3: Change + Apply (Week 9–12)
- Change proposal generation (LLM produces diffs)
- Diff preview UI
- Approve/reject flow
- Apply changes to working copy
- Branch creation
- Commit with message
- Push to remote
- Safety guardrails (protected branches, conflict detection)

### Phase 4: Scale + Team (Future)
- Multi-repo workspaces
- Cross-repo dependency reasoning
- PR draft generation from change proposals
- Team workspaces with shared access
- Codebase health dashboard
- Custom retrieval tuning
- Self-hosted deployment option

## Why phased execution is necessary

Building retrieval, change workflows, and multi-repo simultaneously guarantees all three will be mediocre. Retrieval quality is the foundation — if Q&A answers are bad, nothing else matters. Phase 1 must nail retrieval before Phase 3 touches write operations.

## What must be designed now vs built now

**Designed now, built later:**
- Domain model supports multi-repo (workspace has many repos) — schema ready, UI shows one
- Change proposal entities exist in schema — not populated until Phase 3
- Webhook endpoint route exists — not wired until Phase 2

**Built now:**
- Everything in Phase 0 and Phase 1

## Risks of building too much too early

- Change workflows without solid retrieval = applying bad suggestions
- Multi-repo without stable single-repo = compounding bugs
- UI polish before retrieval works = lipstick on a pig

## User flows

### Flow 1: Connect a repo
1. User signs in via GitHub OAuth
2. User sees workspace with "Connect repository" prompt
3. User selects a repo from their GitHub repos
4. System clones the repo, begins indexing
5. User sees progress indicator
6. Indexing completes → file tree appears, chat becomes active

### Flow 2: Ask a question
1. User types question in chat pane: "How does authentication work?"
2. System retrieves relevant chunks (semantic + lexical + structural)
3. System assembles context, sends to LLM with citation instruction
4. LLM returns answer with file:line citations
5. Answer renders in chat with clickable citation badges
6. User clicks citation → code viewer opens to that file and line

### Flow 3: Browse files
1. User navigates file tree in sidebar
2. User clicks a file → code viewer shows content with syntax highlighting
3. User can ask file-scoped questions: "What does this function do?"

## What makes v1 product-worthy

v1 is not a demo if it can reliably answer questions like:
- "How does the payment flow work?" → Gets a 3-paragraph answer with 5+ file citations
- "What depends on the User model?" → Gets a list of files and symbols that reference it
- "What's the architecture of this repo?" → Gets a generated summary with component descriptions

If the citations are accurate and the answers are grounded, this is already more valuable than pasting files into ChatGPT.

## What v1 must prove

1. **Retrieval quality**: Relevant chunks are found for diverse questions.
2. **Citation accuracy**: Cited files and lines actually contain the referenced information.
3. **Indexing reliability**: Any repo up to ~50K files can be indexed without failure.
4. **Response grounding**: Answers do not hallucinate APIs, functions, or file paths that don't exist.

## Components

### 1. Frontend (Next.js App Router)

**Purpose**: Render the workspace UI — file tree, code viewer, chat, index status.

**Responsibilities**:
- Render three-panel workspace layout
- Manage client state (selected file, conversation, index status)
- Call API endpoints for all data
- Render citations as clickable badges
- Show indexing progress via polling or SSE

**Inputs**: API responses (JSON)
**Outputs**: User interactions → API calls

**Dependencies**: API backend only
**Non-responsibilities**: No direct database access. No GitHub API calls. No LLM calls.

### 2. API Backend (Next.js API Routes)

**Purpose**: Handle all business logic, orchestrate between services.

**Responsibilities**:
- Auth flow (GitHub OAuth callback, session management)
- Workspace CRUD
- Repo connection management
- Trigger ingestion jobs
- Serve file tree and file content
- Orchestrate retrieval for Q&A
- Manage conversations and messages
- Stream LLM responses to frontend

**Inputs**: HTTP requests from frontend
**Outputs**: JSON responses, SSE streams for chat

**Dependencies**: Postgres, Redis/BullMQ, GitHub API, LLM provider, blob storage
**Non-responsibilities**: Does not run long-running indexing inline. Delegates to background worker.

### 3. GitHub Integration Service (module, not separate service)

**Purpose**: All GitHub API interactions.

**Responsibilities**:
- OAuth flow (authorization URL, token exchange, token refresh)
- List user repositories
- Clone repository to blob storage
- Fetch file content via API (for on-demand reads without full clone)
- Read repo metadata (default branch, languages, size)

**Inputs**: GitHub access token, repo identifiers
**Outputs**: Cloned repo path, repo metadata, file content

**Dependencies**: GitHub REST/GraphQL API, blob storage
**Non-responsibilities**: Does not parse files. Does not extract symbols. Does not understand code.

### 4. Ingestion Pipeline (background jobs)

**Purpose**: Transform a cloned repo into indexed, searchable knowledge.

**Responsibilities**:
- Walk the file tree, skip binary/vendored/generated files
- Detect language per file (extension + heuristics)
- Parse files with tree-sitter for symbol extraction
- Produce symbol records (name, kind, file, start_line, end_line, parent)
- Build symbol relationship edges (imports, exports, calls)
- Chunk files with symbol-aware boundaries
- Generate embeddings for each chunk
- Store everything in Postgres/pgvector
- Generate repo summary via LLM
- Report progress to frontend via job status

**Inputs**: Cloned repo path, repo_id
**Outputs**: Populated files, symbols, symbol_relations, chunks, embeddings, repo_summary tables

**Dependencies**: tree-sitter (WASM or native), LLM provider (for summary), pgvector
**Non-responsibilities**: Does not serve queries. Does not handle user interactions.

### 5. Retrieval Engine (module)

**Purpose**: Given a user question + repo context, find the most relevant code.

**Responsibilities**:
- Semantic search: embed question → pgvector nearest-neighbor on chunks
- Lexical search: Postgres full-text search on chunk content
- Structural search: given mentioned symbols, traverse symbol graph for related files
- Rank and merge results from all strategies
- Assemble context window: repo summary + top chunks + file headers + symbol definitions
- Return ranked results with file:line metadata for citation

**Inputs**: Question string, repo_id, optional file_path scope
**Outputs**: Ranked context chunks with metadata (file, start_line, end_line, score, strategy)

**Dependencies**: Postgres/pgvector, symbol graph data
**Non-responsibilities**: Does not call the LLM. Does not generate answers.

### 6. LLM Orchestration Layer (module)

**Purpose**: Turn retrieved context + question into a grounded, cited answer.

**Responsibilities**:
- Construct system prompt with citation instructions
- Assemble retrieved context into LLM prompt
- Call LLM provider (streaming)
- Parse citations from LLM response
- Validate citations against actual repo content
- Return structured answer with validated citations

**Inputs**: Question, assembled context chunks, conversation history
**Outputs**: Answer text with citation objects (file, start_line, end_line, snippet)

**Dependencies**: LLM provider (Anthropic Claude or OpenAI), retrieval results
**Non-responsibilities**: Does not retrieve. Does not store conversations.

### 7. Change Proposal Engine (stubbed in v1)

**Purpose**: Generate, preview, and manage code changes.

**Phase 3 responsibilities** (not built in v1):
- Generate diffs from natural language instructions
- Preview changes in UI
- Apply changes to working copy
- Branch/commit/push operations

**v1 status**: Schema entities exist. No implementation. API returns "not available yet".

### 8. Data Stores

**Postgres + pgvector**:
- All relational data (users, workspaces, repos, files, symbols, conversations, messages)
- Vector embeddings in pgvector with HNSW index
- Full-text search via GIN index

**Redis**:
- BullMQ job queue for background indexing
- Session store
- Cache for frequently accessed repo metadata

**Blob storage (local filesystem in v1, S3-compatible later)**:
- Cloned repository files
- Generated artifacts (repo summaries, analysis reports)

### 9. Background Worker

**Purpose**: Process long-running jobs outside request cycle.

**Responsibilities**:
- Process ingestion jobs (clone → parse → embed → store)
- Process re-index jobs
- Process summary generation jobs
- Report job progress to Redis for frontend polling

**Dependencies**: BullMQ, Postgres, blob storage, tree-sitter, LLM provider
**Non-responsibilities**: Does not handle HTTP requests.

### 10. Auth + Workspace Manager

**Purpose**: User identity and workspace isolation.

**Responsibilities**:
- GitHub OAuth flow
- Session management (JWT or server-side sessions)
- Workspace creation and ownership
- Repo-to-workspace association
- Access control (user can only access their workspaces)

**Dependencies**: GitHub API, Postgres, Redis (sessions)
**Non-responsibilities**: Does not manage repo content. Does not handle indexing.

# 06_RETRIEVAL_AND_REASONING_DESIGN

## Multi-strategy retrieval

RepoBrain must never rely on a single retrieval strategy. Each strategy catches different kinds of relevance:

### 1. Semantic retrieval (pgvector)
- Embed the user question with the same model used for chunks
- k-NN search on pgvector HNSW index (k=20)
- Returns chunks most semantically similar to the question
- **Good for**: conceptual questions ("how does auth work?"), finding related code
- **Bad for**: exact name lookups, structural queries

### 2. Lexical retrieval (Postgres FTS)
- Full-text search on chunk content using `ts_vector` + `ts_query`
- Also trigram search (`pg_trgm`) for partial matches
- Returns chunks containing the exact or near-exact terms
- **Good for**: "where is UserService defined?", exact symbol names
- **Bad for**: conceptual questions with no keyword overlap

### 3. Structural retrieval (symbol graph)
- Extract mentioned symbols from the question (via regex + known symbol list)
- Look up Symbol records matching those names
- Traverse SymbolRelation edges (imports, calls, extends) 1–2 hops
- Return the chunks containing those symbols and their related symbols
- **Good for**: "what depends on X?", "what does Y call?", dependency questions
- **Bad for**: questions with no symbol references

### Rank and merge

1. Run all three strategies in parallel
2. Each returns scored results (0.0–1.0)
3. Normalize scores within each strategy
4. Combine with weights: semantic=0.45, lexical=0.30, structural=0.25
5. Boost if a chunk appears in multiple strategies (intersection bonus: +0.15)
6. Deduplicate overlapping chunks (prefer the higher-scored one)
7. Take top-K results (K=15 for standard, K=25 for deep questions)

### Context assembly

The final LLM prompt is assembled as:

```
[System prompt with citation instructions]

## Repository Overview
{repo_summary.summary_text}

## Relevant Code (ordered by relevance)

### File: src/auth/middleware.ts (lines 15-45)
```typescript
{chunk content}
```

### File: src/models/user.ts (lines 1-30)
```typescript
{chunk content}
```

[... top-K chunks ...]

## Conversation History
{last 5 messages if any}

## Question
{user question}
```

### File-scoped context

When the user asks about a specific file:
1. Include the entire file content (up to 500 lines; chunk if larger)
2. Include symbols defined in that file
3. Include files that import from this file (via SymbolRelation)
4. Still run semantic search for additional context
5. Weight file-scoped results higher (+0.3 boost)

### Citation strategy

The LLM system prompt instructs:
```
When you reference code, always cite the source using this format: [file:path/to/file.ts:L15-L30]

Rules:
- Every factual claim about the code MUST have at least one citation.
- Only cite files and line ranges that appear in the provided context.
- If you cannot find evidence for a claim in the context, say "I don't have enough context to answer this specifically."
- Never invent file paths or line numbers.
```

After LLM response:
1. Parse all `[file:...:L..-L..]` citations
2. Validate each against actual repo: file exists? lines exist? content roughly matches?
3. Invalid citations get flagged (not silently removed — the user should know)
4. Valid citations become clickable in the UI

### Failure handling

When retrieval returns low-confidence results (all scores < 0.3):
- The system tells the user: "I found limited relevant code for this question. Here's what I found, but the answer may be incomplete."
- Never fabricate an answer when context is insufficient.

When the question is clearly outside the repo's domain:
- "This question doesn't seem to be about the codebase. RepoBrain answers questions grounded in your repository."

### How to avoid generic answers

1. The system prompt explicitly forbids generic programming advice
2. Every answer must reference at least one specific file from the repo
3. If the LLM's answer contains no citations, the system appends: "Note: I wasn't able to ground this answer in specific code. Consider rephrasing your question."
4. Post-processing validates citation density: answers with < 1 citation per 100 words get a warning

# 07_CHANGE_WORKFLOW_CONTRACT

> **This section defines the Phase 3 contract. It is NOT built in v1, but the schema and state model must be understood now to avoid painting into a corner.**

## State machine

```
                    ┌──────────┐
                    │ proposed │ (LLM generates diff)
                    └────┬─────┘
                         │ user reviews
                    ┌────▼─────┐
              ┌─────│ previewed│ (diff shown in UI)
              │     └────┬─────┘
              │          │ user clicks "Approve" or "Reject"
         ┌────▼───┐ ┌───▼────┐
         │rejected│ │approved│
         └────────┘ └───┬────┘
                        │ system applies to working copy
                   ┌────▼───┐
                   │ applied│
                   └───┬────┘
                       │ user clicks "Create Branch & Commit"
                  ┌────▼─────┐
                  │committed │
                  └────┬─────┘
                       │ user clicks "Push"
                  ┌────▼──┐
                  │ pushed│
                  └───────┘
```

## Safety requirements

1. **Propose ≠ Apply.** Generating a diff does not modify any files. Ever.
2. **Apply is reversible.** Applied changes can be reverted before commit.
3. **Branch is mandatory.** Changes are never committed to the default branch directly.
4. **User confirms every transition.** No auto-apply, no auto-push.
5. **Conflict detection.** If the base file changed since the diff was generated, the change cannot be applied without regeneration.
6. **Scope lock.** A change proposal targets specific files. It cannot silently modify other files.

## UX expectations

- The diff preview must look like a standard unified diff (green/red lines)
- The user must be able to see before/after side-by-side
- Each proposed change shows which conversation message generated it
- Multiple changes can be batched into one commit
- The branch name is auto-suggested but user-editable

# 09_MODULE_BOUNDARIES_AND_INTERFACE_CONTRACTS

## Module: `github`

**Owns**: All GitHub API communication, OAuth flow, repo cloning
**Depends on**: Configuration (client ID, secret), blob storage path
**Must not mutate**: Database records (reports results, caller writes)
**Exposes**:
```typescript
getAuthorizationUrl(state: string): string
exchangeCodeForToken(code: string): Promise<GitHubToken>
listUserRepos(token: string): Promise<GitHubRepo[]>
cloneRepo(token: string, owner: string, name: string, targetPath: string): Promise<void>
getRepoMetadata(token: string, owner: string, name: string): Promise<RepoMetadata>
```
**Assumptions**: Token is valid and has repo scope.

## Module: `ingestion`

**Owns**: File parsing, symbol extraction, chunking, embedding generation
**Depends on**: `github` (for cloned repo path), Postgres, pgvector, tree-sitter, LLM provider
**Must not mutate**: GitHub state, workspace/user records, conversations
**Exposes**:
```typescript
indexRepo(repoConnectionId: string, clonePath: string): Promise<IndexResult>
getIndexProgress(jobId: string): Promise<IndexProgress>
```
**Assumptions**: Repo is already cloned to the path. Database tables exist.

## Module: `retrieval`

**Owns**: Search across semantic, lexical, structural strategies. Ranking and context assembly.
**Depends on**: Postgres/pgvector, chunk/symbol/file data
**Must not mutate**: Any data. Pure read-only.
**Exposes**:
```typescript
retrieve(question: string, repoConnectionId: string, options?: RetrievalOptions): Promise<RetrievalResult>
// RetrievalResult = { chunks: RankedChunk[], repoSummary: string, strategies_used: string[] }
// RankedChunk = { content, filePath, startLine, endLine, score, strategy }
```
**Assumptions**: Repo is indexed. Embeddings exist.

## Module: `llm`

**Owns**: LLM API calls, prompt construction, citation parsing, response streaming
**Depends on**: LLM provider API (Anthropic/OpenAI), retrieval results
**Must not mutate**: Database records (caller stores messages)
**Exposes**:
```typescript
generateAnswer(question: string, context: RetrievalResult, history: Message[]): AsyncGenerator<AnswerChunk>
// AnswerChunk = { type: 'text', content: string } | { type: 'citation', citation: Citation }
```
**Assumptions**: Context is already retrieved and assembled.

## Module: `workspace`

**Owns**: User, workspace, repo connection CRUD. Auth session management.
**Depends on**: Postgres, `github` (for OAuth)
**Must not mutate**: File/symbol/chunk data, conversations
**Exposes**:
```typescript
createWorkspace(userId: string, name: string): Promise<Workspace>
connectRepo(workspaceId: string, githubRepoId: string): Promise<RepoConnection>
getWorkspace(workspaceId: string): Promise<WorkspaceWithRepos>
```

## Module: `chat`

**Owns**: Conversation and message CRUD. Orchestrates retrieval → LLM → store.
**Depends on**: `retrieval`, `llm`, Postgres
**Must not mutate**: Repo data, workspace config
**Exposes**:
```typescript
askQuestion(conversationId: string, question: string): AsyncGenerator<AnswerChunk>
getConversation(conversationId: string): Promise<ConversationWithMessages>
listConversations(workspaceId: string): Promise<Conversation[]>
```

## Module: `changes` (Phase 3 stub)

**Owns**: SuggestedChange, BranchAction lifecycle
**Depends on**: `github`, `llm`, Postgres
**Not implemented in v1. Exposes stubs that return 501.**

### WS1: Project Skeleton + Auth (Foundation)

**Dependencies**: WS1

**Dependencies**: WS2 (needs cloned repo)

- Embedding generation (batched, with rate limiting)
- pgvector storage with HNSW index
- Full-text search index (GIN)
- Index job progress tracking
- UI: index progress indicator

### WS4: Retrieval Engine
**Dependencies**: WS3 (needs indexed data)
**Deliverables**:
- Semantic search function (embed query → pgvector k-NN)
- Lexical search function (Postgres FTS)
- Structural search function (symbol graph traversal)
- Rank-and-merge algorithm
- Context assembly function
- Retrieval options (scope to file, depth control)
- Unit tests with fixture data

### WS5: LLM + Chat
**Dependencies**: WS4 (needs retrieval), WS1 (needs auth)
**Deliverables**:
- LLM provider abstraction (start with Anthropic Claude)
- System prompt with citation instructions
- Prompt construction from retrieval results
- Streaming response handler
- Citation parser + validator
- Conversation + message persistence
- Chat API (ask question, list conversations, get conversation)
- UI: chat pane, message rendering, citation badges, streaming indicator

### WS6: File Tree + Code Viewer
**Dependencies**: WS2 (needs repo data), WS3 (needs file records)
**Deliverables**:
- File tree API (returns nested structure from File records)
- File content API (reads from blob storage)
- UI: collapsible file tree with search
- UI: code viewer with syntax highlighting (Shiki or Prism)
- UI: citation click → navigate to file:line with highlight
- UI: "Ask about this file" button

### WS7: Repo Summary
**Dependencies**: WS3 (needs indexed data), WS5 (needs LLM module)
**Deliverables**:
- Summary generation job (after indexing completes)
- LLM prompt for architectural summary
- RepoSummary table persistence
- Summary included in retrieval context
- UI: summary view in workspace (optional for v1, nice-to-have)

```
Week 1:  WS1 (skeleton + auth)
Week 2:  WS2 (workspace + repo connection) + WS6 starts (file tree UI)

# 12_RISKS_AND_FAILURE_MODES

## Product risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Retrieval quality is poor | Users get bad answers, lose trust immediately | Multi-strategy retrieval, citation validation, "low confidence" warnings |
| Indexing takes too long for large repos | Users abandon during wait | Progress indicator, allow partial Q&A during indexing, skip vendored/generated files |
| Answers feel generic, not grounded | No differentiation from ChatGPT | Strict citation requirements, refuse to answer without evidence |

## Technical risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| tree-sitter WASM doesn't support a language | Symbols missing for that language | Graceful fallback to regex-based extraction, clearly mark unsupported languages |
| pgvector performance degrades at scale | Slow queries for large repos | HNSW index tuning, limit repo size in v1, monitor query latency |
| Embedding API rate limits | Indexing stalls | Batched requests with backoff, progress resumes from last checkpoint |
| GitHub token expires mid-clone | Clone fails silently | Token refresh before clone, explicit error on auth failure |

## Retrieval risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Semantic search returns irrelevant results | Bad answers | Combine with lexical + structural, require minimum score threshold |
| Chunking splits a function across two chunks | Incomplete context | Symbol-aware chunking (never split mid-symbol), overlap between chunks |
| Repo summary is inaccurate | Misleading context in every query | Validate summary against actual file structure, regenerate periodically |

## UX risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Citation links break after re-index | User clicks dead link | Citations store content_hash, detect stale citations |
| Chat feels slow | Poor experience | Stream responses, show retrieval phase indicator |
| File tree is overwhelming for large repos | User can't find files | Search/filter in file tree, collapse by default |

## Scope risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Team builds change workflow before retrieval is solid | Bad suggestions applied to code | Phase gate: retrieval quality must pass acceptance criteria before Phase 3 begins |
| Multi-repo creeps into v1 | Complexity explosion | Schema supports it, code enforces single repo per workspace in v1 |

## Data consistency risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Re-index runs while queries are executing | Stale/mixed results | Index to new snapshot, swap atomically (new commit SHA only updates on completion) |
| Partial index failure | Incomplete data | Index job is all-or-nothing per file; failed files are retried, not skipped |

# 13_DECISION_LOG

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **Next.js monorepo** (not separate frontend + backend) | Reduces deployment complexity for v1. API routes colocate with frontend. Split later if needed. |
| 2 | **Postgres + pgvector** (not Pinecone/Weaviate) | One database for relational + vector data. Simpler ops. pgvector is mature enough for v1 scale. |
| 3 | **BullMQ + Redis** (not SQS/RabbitMQ) | Lightweight, Node-native, good DX. Redis already needed for sessions. |
| 4 | **tree-sitter WASM** (not Language Server Protocol) | LSP is heavy and requires per-language servers. tree-sitter WASM runs in-process, supports many languages with one approach. |
| 5 | **Symbol-aware chunking** (not fixed-size) | Chunks that split mid-function produce bad retrieval. Symbol boundaries produce coherent chunks. |
| 6 | **Three-strategy retrieval** (not semantic-only) | Semantic alone misses exact matches. Lexical alone misses conceptual queries. Structural is needed for dependency questions. |
| 7 | **Citations as structured objects** (not inline markdown) | Structured citations enable validation, click-to-navigate, and staleness detection. |
| 8 | **Single repo per workspace in v1** (schema allows multi) | Keeps v1 simple. Schema is ready for multi-repo without migration. |
| 9 | **Blob storage for cloned repos** (not re-fetch from GitHub per request) | Faster file access, works offline, reduces GitHub API calls. |
| 10 | **Anthropic Claude as primary LLM** (not OpenAI) | Better instruction following for citation format compliance. Can switch via abstraction layer. |
| 11 | **Drizzle ORM** (not Prisma) | Lighter, SQL-closer, better pgvector support, no binary engine. |
| 12 | **Local filesystem blob storage in v1** (S3 later) | No cloud dependency for local dev. Abstraction layer makes S3 swap trivial. |

- Before writing code, identify which module boundary you're working in (§09)

# 15_EXECUTIVE_SUMMARY_FOR_HUMAN_FOUNDER

## What RepoBrain v1 actually is

A web app where you connect a GitHub repo, it indexes everything (files, functions, classes, imports), and you can ask it questions about your codebase and get answers grounded in actual code with clickable citations. Think "ChatGPT but it actually knows your repo and can prove it."

## What to build first

1. Auth + workspace shell (week 1)
2. GitHub connection + clone (week 2)
3. Indexing pipeline — parse, extract symbols, embed (week 3-4)
4. Search + Q&A with citations (week 5)
5. File viewer with citation navigation (week 6)

## What to delay

- Code changes / editing / commit / push (Phase 3)
- Multiple repos per workspace (Phase 4)
- PR generation (Phase 4)
- Self-hosted option (much later)

## What makes it valuable

If the answers are grounded and the citations are accurate, this immediately beats "paste files into ChatGPT." The persistent index means it gets better as you use it, and the file viewer creates a tight loop between question → answer → code → deeper question.

## What would most likely kill the project if done wrong

1. **Bad retrieval.** If answers aren't grounded in the right code, the product is useless. The three-strategy retrieval (semantic + lexical + structural) is the most critical engineering work.
2. **Slow indexing.** If it takes 30 minutes to index a repo, users leave. Must be under 5 minutes for a typical repo (< 10K files).
3. **Hallucinated citations.** If the system cites files that don't contain what it claims, users lose trust permanently. Citation validation is non-negotiable.
4. **Scope creep into Phase 3.** Building change workflows before retrieval is solid guarantees applying bad suggestions to real code. Resist the urge.
