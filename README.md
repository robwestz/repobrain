# RepoBrain

> A third-person intelligence layer for your codebase. Ask questions, get sourced answers, navigate at the speed of thought.

RepoBrain ingests a Git repository, indexes it across three retrieval strategies (semantic, lexical, structural), and exposes 15 features for understanding, navigating, and reasoning about code at scale. Every answer cites real `file:line` ranges in the indexed source — no hallucinated paths, no made-up signatures.

Built as a single TypeScript application: Next.js 15 + Drizzle ORM + Postgres+pgvector + BullMQ + Redis + tree-sitter. One language, one process model, one deploy.

## Features

| # | Feature | What it does |
|---|---------|--------------|
| 1 | **Code Conversations** | Ask questions about your codebase in natural language. Multi-strategy retrieval with cited answers. File-scoped or repo-wide context. |
| 2 | **Natural Language Search** | Structured code search: "find error handlers without retry", "show me singleton patterns", "where is User defined". Intent classification + heuristic routing. |
| 3 | **Anchored Discussions** | Persistent threads attached to specific code regions. Threads anchor to symbols so they survive line-shift refactors. |
| 4 | **Semantic Bookmarks** | Save annotated code regions with notes. Survives refactors. Searchable. |
| 5 | **Git Timeline** | Semantic visualization of repo history. See features emerge across commits. |
| 6 | **Code Health Dashboard** | Per-language LOC, complexity, dependency stats, test coverage signals. |
| 7 | **API Surface Map** | All HTTP endpoints (Next.js, Express, FastAPI, etc.) with auth, methods, params extracted. |
| 8 | **Pattern Detective** | Find recurring patterns (or anti-patterns) across the codebase. |
| 9 | **Smart Onboarding** | Generated walkthroughs tailored to a role (frontend dev, backend dev, ops). |
| 10 | **Living Architecture Diagrams** | Auto-generated Mermaid diagrams of system structure. Updates on re-index. |
| 11 | **Codebase Narrator** | Step-by-step narrated tours of how a feature works, generated by an LLM. |
| 12 | **Blast Radius** | "If I change X, what breaks?" — symbol relation traversal. |
| 13 | **Dependency Galaxy** | Force-directed visualization of module dependencies. |
| 14 | **What-If Sandbox** | Hypothetical change analysis without touching the actual code. |
| 15 | **Cross-Repo Intelligence** | Detect shared types, copy-paste patterns, and contract drift across multiple repos in a workspace. |

### DeepWiki Integration (added Mar 2026)

- **Domain context partitioning** — 6 domains (security, frontend, backend, db, config, test) for scoped retrieval
- **Specialist agents** — Code Review, Security Audit, ADR Generator (each with dedicated UI)
- **Deep Research** — iterative multi-turn investigation with gap analysis
- **Multi-provider LLM** — JSON-based config for OpenAI, Anthropic, and Ollama
- **Model selector** in workspace header

## Quick Start (local development)

### Prerequisites

- Node.js 20+
- Docker Desktop (for local Postgres + Redis)
- A GitHub OAuth App ([create one](https://github.com/settings/developers)) — set callback to `http://localhost:3000/api/auth/github/callback`
- An OpenAI or Anthropic API key

### Setup

```bash
# 1. Clone and install
git clone https://github.com/robwestz/repobrain.git
cd repobrain/src
npm install

# 2. Start Postgres (with pgvector) and Redis locally
docker-compose up -d

# 3. Create .env from template
cp .env.example .env
# Edit .env — fill in GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET, OPENAI_API_KEY (or ANTHROPIC_API_KEY), and SESSION_SECRET
# For local Docker, set:
#   DATABASE_URL=postgresql://repobrain:repobrain@localhost:5432/repobrain
#   REDIS_URL=redis://localhost:6379

# 4. Run database migrations
npm run db:migrate

# 5. Download tree-sitter grammars (one-time)
npm run grammars:download

# 6. Start the dev server
npm run dev

# 7. In a separate terminal, start the workers
npm run worker
```

Open [http://localhost:3000](http://localhost:3000), log in with GitHub, create a workspace, connect a repo, wait for indexing to complete, and start asking questions.

### Production environment

Set the variables in `.env.production.template` and deploy. The app fails fast in production if any required secret is missing — see `src/lib/env.ts`.

```
│        │                                                             │

│        ├──→ llm       ─→ multi-provider (OpenAI, Anthropic, Ollama)  │
│        ├──→ specialists ─→ code-review, security-audit, adr-gen     │
│        ├──→ chat/deep-research ─→ iterative multi-turn investigation │
│        └──→ ingestion ─→ walker + symbols + relations + chunker      │
│        │                                                             │
│        ▼                                                             │
│   lib/db (Drizzle) ─→ Postgres + pgvector                           │
│   lib/redis        ─→ Redis (BullMQ queues)                         │
└───────────────────────┬──────────────────────┬──────────────────────┘
                        │                      │
                        ▼                      ▼
              ┌──────────────────┐   ┌──────────────────┐
              │ BullMQ Workers   │   │  Postgres        │
              │  • clone.worker  │   │  + pgvector      │
              │  • ingest.worker │   │                  │
              └──────────────────┘   └──────────────────┘
```

### Layered conventions

| Layer | Purpose | Examples |
|-------|---------|----------|
| `app/api/*` | HTTP boundaries — validate input, call modules, format response | `chat`, `workspaces`, `github` |
| `modules/*` | Business logic — pure-ish, no HTTP concerns | `retrieval`, `llm`, `chat`, `ingestion` |
| `lib/*` | Cross-cutting infrastructure | `db`, `redis`, `auth`, `env`, `logger`, `rate-limit` |
| `workers/*` | Long-running background jobs | `clone.worker`, `ingest.worker` |
| `components/*` | React UI components | `chat/`, `code-viewer/`, `specialists/` |

API routes are thin: they handle auth, parse params, call a module function, return JSON or stream SSE. All real work lives in `modules/`.

## Configuration

### Required environment variables

| Variable | Description |
|----------|-------------|
| `GITHUB_CLIENT_ID` | GitHub OAuth app client ID |
| `GITHUB_CLIENT_SECRET` | GitHub OAuth app client secret |
| `GITHUB_REDIRECT_URI` | OAuth callback URL (must match GitHub app config) |
| `DATABASE_URL` | PostgreSQL connection string (with `pgvector` extension installed) |
| `REDIS_URL` | Redis connection string |
| `OPENAI_API_KEY` | OpenAI API key (or use Anthropic) |
| `ANTHROPIC_API_KEY` | Anthropic API key (alternative to OpenAI) |
| `DEFAULT_LLM_PROVIDER` | `openai` or `anthropic` |
| `SESSION_SECRET` | 32+ char hex string (`openssl rand -hex 32`) |
| `BLOB_STORAGE_PATH` | Where cloned repos are stored locally |
| `NEXT_PUBLIC_APP_URL` | Public URL of the app |

### LLM provider config

Edit `src/config/llm-providers.json` to add or modify available models. The model selector in the workspace header reads from this file.

### Multi-provider — when to use what

| Provider | Best for | Notes |
|----------|----------|-------|
| **OpenAI** (default) | General Q&A, fast iteration | `gpt-4o-mini` for retrieval-grounded answers |
| **Anthropic** | Long context, complex reasoning, ADR generation | `claude-sonnet-4-6` for specialists |
| **Ollama** | Local / private / offline | Requires Ollama server running |

Production hardening enforces fail-fast in `NODE_ENV=production`: missing secrets crash the process at startup, not at first request.

## How Retrieval Works

When a user asks a question, the chat module runs three retrieval strategies in parallel:

| Strategy | What it does | Backed by |
|----------|--------------|-----------|
| **Semantic** | Vector similarity search over code chunks | `pgvector` HNSW index |
| **Lexical** | Full-text keyword search | Postgres `tsvector` |
| **Structural** | Symbol relation traversal (calls, uses, imports) | Drizzle relations on `symbols` + `symbolRelations` |

The results are merged, ranked, and passed to a context assembler that builds an LLM-ready window respecting the token budget. Domain partitioning (security/frontend/backend/db/config/test) can further scope retrieval to relevant directories.

The LLM is instructed to **only cite what retrieval returned**. Citations reference real `file:line` ranges. This is enforced by the prompt and verified by the citation parser.

## Indexing Pipeline

When a repo is connected:

1. **Clone worker** (`clone.worker.ts`) — `git clone` to `BLOB_STORAGE_PATH/{workspaceId}/{repoId}/`
2. **Ingest worker** (`ingest.worker.ts`) runs the pipeline:
   - **Walker** — traverse the filesystem, respect `.gitignore`, identify text files
   - **Language detector** — map extensions to tree-sitter grammars
   - **Symbol extractor** — parse with tree-sitter, extract definitions + signatures
   - **Relation builder** — calls, uses, imports between symbols
   - **Chunker** — split files into overlapping chunks aligned to symbol boundaries
   - **Embedder** — generate vector embeddings for each chunk
   - **DB writer** — bulk insert files, symbols, relations, chunks, embeddings

Workers report progress via Postgres + WebSocket polling. The frontend shows indexing progress in real time.

## Development Notes

### Adding a new feature

The conventions in `.planning/ORCHESTRATION.md` were designed for parallel agent execution. They still apply for human work:

1. Create a new directory under `src/modules/{feature}/`
2. Add a schema file `src/lib/db/schema-{feature}.ts` and append `export * from "./schema-{feature}"` to `schema.ts`
3. Create the API route at `src/app/api/workspaces/[workspaceId]/repos/[repoId]/{feature}/route.ts`
4. Create the page at `src/app/workspace/[workspaceId]/{feature}/page.tsx`
5. Create components at `src/components/{feature}/`
6. Add a nav item to `src/components/layout/sidebar-nav.tsx`
7. Run `npm run db:generate` to create the migration
8. Run `npm run db:migrate` to apply it
9. Run `npm run ci` before committing

### Style conventions

- **TypeScript strict mode** — no `any`, no ignored errors
- **No `console.log`** in production code — use `logger` from `@/src/lib/logger`
- **Auth in every API route** — `await getSession()` then verify ownership before any DB read
- **Tables over prose** in documentation
- **Imperative commit messages** — `feat:`, `fix:`, `docs:`, `refactor:` prefixes

### Build hardening

`next.config.ts` pins `outputFileTracingRoot` to the project directory to prevent Next.js from inferring the wrong workspace root when multiple `package-lock.json` or `pnpm-lock.yaml` files exist elsewhere on the system. This eliminates the intermittent ENOENT during trace collection on Windows.
